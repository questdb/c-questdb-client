# QWP/WebSocket Java/Rust golden payload fixture

Date: 2026-04-28

Status: validation note for the Step 3 Java/Rust replay-payload parity gate in
`doc/QWP_WEBSOCKET_VALIDATION_PLAN.md`.

## Scope

Added a Rust test fixture:

```text
tests::qwp_ws_java_golden::qwp_ws_replay_payloads_match_java_golden_bytes
```

The fixture compares exact unmasked QWP application payload bytes generated by
the Java client and the Rust replay encoder for the same logical batches.

The current fixture covers:

- replay-mode WebSocket QWP payload bytes, not masked WebSocket frames
- full schema mode
- disabled schema references
- dense symbol dictionary prefix from global id `0`
- a later replay batch that references only the highest known symbol id
- `SYMBOL`, `LONG`, `DOUBLE`, and designated `TIMESTAMP_NANOS` columns
- repeated table/schema shape across batches

It does not yet cover arrays, decimals, UTF-8 string payloads, sparse columns, or
schema evolution. Those should be added before treating Java/Rust parity as
complete for the whole QWP type surface, but they are not needed to start the
volatile queue prototype.

## Reflection

This closes the narrow Java/Rust parity question that blocked Step 5. The
fixture is intentionally small and payload-level: it does not lock Rust to
Java's sender API, threading model, WebSocket masking, or connection behavior.

The next step can proceed to the volatile queue prototype, while keeping
extended-type golden fixtures as a follow-up compatibility hardening task.
